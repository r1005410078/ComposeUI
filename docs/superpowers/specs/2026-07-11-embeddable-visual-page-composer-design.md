# ComposeUI 可嵌入式可视化页面编排引擎设计

## 目标

构建一个可嵌入任意 Web 项目的桌面端可视化页面编辑器。宿主应用可以只为指定的独立页面启用编辑能力，将已有业务组件与宿主提供的数据组合为 JSON 页面文档，并在运行时渲染。

编辑器内核不依赖宿主框架，Vue 和 React 是首批官方适配器。产品并非通用的 Figma 替代品，首要工作流是组装可复用的业务组件和数据驱动页面。高级矢量创作能力作为可选扩展，不阻塞核心产品交付。

## 产品范围

核心版本包含：

- Unity 风格的组件树、舞台、组件面板以及属性和数据绑定面板。
- 组件树、舞台选区、属性编辑器和运行时渲染器共用的一棵节点树。
- 自由布局、自动布局和 Grid 容器节点。
- 拖放、嵌套、排序、多选、锁定、显隐、对齐、缩放/平移以及撤销/重做。
- 内置基础图形和 SVG 展示。
- 由宿主应用显式注册已有 Vue 和 React 业务组件。
- 将选中的节点子树创建为可复用自定义组件，并以类似 Prefab 的定义和实例形式使用。
- JSON 持久化，以及解析已注册组件、数据源、状态和动作的宿主运行时。
- 页面变量、宿主提供的接口结果和宿主提供的全局状态绑定。
- 可控的挂载/卸载、只读模式、保存回调、错误回调，以及同一页面中多个编辑器实例的隔离。
- 基于 Yjs 的多人实时协同、离线编辑、远程选区和本地撤销/重做。

核心版本不包含响应式断点、移动端编辑、内置 HTTP/鉴权/缓存管理、任意 JavaScript 执行以及完整的矢量路径编辑。

## 可嵌入架构

产品按职责拆分为独立包，宿主只安装所需能力：

- `@composeui/core`：文档 Schema、命令系统、布局协议、绑定协议和校验，不包含 UI 和框架运行时。
- `@composeui/editor`：组件树、无限工作区（Workspace）、页面画板（Artboard）、组件面板和属性面板。
- `@composeui/runtime`：框架无关的页面解析、绑定解析和组件适配协议。
- `@composeui/react`、`@composeui/vue`：React 和 Vue 的编辑器挂载及组件运行时适配器。
- `@composeui/figma-import`：第三阶段按需安装的 Figma 导入模块。
- `@composeui/collaboration`、`@composeui/collaboration-yjs`：协同端口与 Yjs 官方实现，按需安装。

编辑器内部采用事务型微内核、规范化 Record Store、Command/Query/Projection、插件贡献系统和编译器流水线。完整架构见：[事务型编辑器微内核架构设计](./2026-07-11-transactional-editor-microkernel-architecture-design.md)。

宿主通过受控 API 创建编辑器实例。核心配置至少包括：

```ts
createEditor({
  container,
  componentMountRoot,
  componentPortalRoot,
  document,
  components,
  hostContext,
  adapters,
  dataSources,
  globalState,
  actions,
  assetAdapter,
  collaboration,
  readOnly,
  onChange,
  onSave,
  onError
})
```

编辑器实例提供 `mount`、`updateDocument`、`setReadOnly`、`save` 和 `unmount` 生命周期。文档默认由宿主控制；编辑器通过 `onChange` 上报变更和脏状态，通过 `onSave` 请求宿主持久化，不自行决定接口、路由或存储方式。

所有选区、命令历史、快捷键、剪贴板状态、组件注册表、Portal 容器和资源上下文都属于编辑器实例，禁止依赖进程级全局单例。宿主可以在同一页面创建多个互不影响的编辑器实例。

### 样式隔离约束

Shadow DOM 只隔离编辑器自身 UI，不隔离宿主项目中的业务组件样式。编辑器工具栏、组件树、属性面板、选区和编辑器弹层挂载在 `container` 的 Shadow Root；业务组件挂载在 Light DOM 的 `componentMountRoot`，继续使用宿主项目的全局 CSS、CSS Modules、主题变量和组件库样式。

业务组件产生的下拉框、对话框、Tooltip 等弹层统一挂载到 Light DOM 的 `componentPortalRoot`；编辑器自己的弹层保留在 Shadow Root 内。选区 SVG 覆盖层通过统一坐标变换和业务组件的边界信息保持对齐。编辑器的 reset 样式不得作用于 Light DOM，内部 CSS 变量统一使用 `--composeui-*` 前缀。

快捷键和剪贴板处理只在实例获得焦点时生效；全局事件监听必须在 `unmount` 时释放。对于无法隔离宿主 CSS 或具有副作用的业务组件，可选择 iframe 预览适配器。

`ComponentAdapter` 是框架无关的扩展点，负责组件挂载、更新、卸载、属性元数据和事件桥接。Vue/React 只是官方实现；Angular、Svelte、Web Components 或原生组件可以后续通过相同协议接入。

### 上下文注入约束

每个画板为选定框架创建一个页面级运行时根，上下文只在该根部注入一次，禁止为每个业务组件分别创建应用实例或重复注入。`hostContext` 统一承载宿主暴露的 `dataSources`、`globalState`、`actions`、Router、i18n、主题和业务服务，再由当前 `ComponentAdapter` 桥接到框架机制：React 使用 Provider Root，Vue 使用 App Root 的插件和 `provide`，Angular 使用共享的 `EnvironmentInjector`。

同一框架的业务组件、Portal 和 Teleport 必须属于该页面级运行时根，从而继承相同上下文和生命周期。宿主已有的 Store、Router、主题或服务实例必须显式传给适配器；新建的框架根不得假定能够自动继承宿主框架上下文。

### 单框架约束

每个画板必须在 `PageDocument.runtime.adapterId` 中声明唯一业务组件框架，例如 `vue`、`react` 或 `angular`。一个画板内只允许该 Adapter 注册的业务组件和框架无关基础节点，不支持跨框架业务组件共存或嵌套。

组件面板按当前 `adapterId` 过滤业务组件；自定义组件定义也必须属于同一 Adapter。已有业务组件后不能直接切换 Adapter，只能执行显式的页面迁移：保留框架无关节点、尝试映射业务组件，并为无法转换的节点生成迁移报告。

## 用户体验

组件树参考 Unity Hierarchy：页面是根节点，父子关系决定组合关系和逻辑顺序。在组件树中选中节点时，舞台同步选中；在舞台中选中元素时，组件树同步定位对应节点。完整的节点类型、拖拽、状态、实例、搜索、键盘操作和验收规则见：[Unity 风格组件树设计](./2026-07-11-unity-style-component-tree-design.md)。

组件面板包含内置节点、宿主注册组件和自定义组件定义。用户将条目拖放到舞台或兼容的容器中。属性面板展示静态属性、已声明的绑定目标、已声明事件、布局控制以及节点显隐和锁定控制。

编辑业务组件内容与编辑舞台是两个独立模式。在舞台编辑模式中，画布接管指针事件，用于选择、拖拽、调整尺寸和嵌套；在交互预览模式中，DOM 渲染的业务组件接收输入事件。

## 无限工作区与页面画板

无限工作区（Workspace）参考 Unity Scene View；页面画板（Artboard）是宿主最终渲染的独立页面。两者必须分离：平移、缩放、辅助网格和编辑上下文属于无限工作区，页面尺寸、背景、内容树和溢出规则属于画板。

- 无限工作区支持无限平移和以鼠标位置为中心的连续缩放；画板是世界坐标中有明确宽高的页面根节点。
- 画板配置页面尺寸、背景和 `visible/hidden/scroll` 溢出策略。画板外的临时对象不进入运行时文档；保存前必须移动到画板内或明确删除。
- 背景显示可随缩放密度变化的辅助网格、坐标原点和标尺。网格用于观察和吸附，不会自动将自由布局节点限制为格子单位。
- 用户可点击选中节点、拖出框选区域多选、按住修饰键增减选区，并在组件树、画布和属性面板间保持同步。
- 选中节点显示边界框、八个缩放控制点和旋转控制点；多选时显示共同边界框。拖动、缩放、旋转和对齐都通过命令系统修改文档。
- 画布提供网格吸附、对象边缘/中心吸附、间距辅助线和对齐辅助线；用户可以临时关闭吸附。
- 拖动节点经过容器时显示可放置高亮。放入自由、Auto Layout 或 Grid 容器时，编辑器转换或初始化该节点对应的布局数据。
- 自由布局节点在画布上按照世界坐标渲染；Auto Layout 和 Grid 的子节点在容器内部由布局引擎计算位置。画布只负责展示计算结果和提供编辑控制。
- 业务组件由 DOM 真实渲染，并使用与无限工作区相同的平移、缩放和世界坐标变换。选中业务组件时，SVG 覆盖层显示选区和控制柄；在交互预览模式中才将指针事件交给组件本身。
- 画布必须支持从根节点到当前容器/自定义组件定义的编辑上下文切换，并通过面包屑显示当前位置。

## 文档模型

`PageDocument` 是唯一的持久化数据源，包含文档元数据、`runtime.adapterId`、画板配置、页面根节点、已注册数据源引用、自定义组件定义、资源清单、导入来源信息和 Schema 版本。无限工作区的视口位置、缩放比例和临时对象属于编辑器会话，不进入运行时页面文档。

`Node` 包含 id、类型、适用时的子节点、样式、局部变换、布局配置、静态属性、结构化属性绑定、事件动作引用、显隐和锁定标志，以及可选的导入来源信息。

核心版本的节点公共视觉模型支持：

- 单填充、单描边、独立圆角、透明度和单层阴影。
- 位置、尺寸、旋转、二维变换矩阵、裁剪和父子局部坐标。
- 字体族、字重、字号、行高、字距、段落和文本对齐。
- 图片资源引用、SVG、矢量路径以及资源适配模式。
- 原始来源、外部节点 id、导入时间和转换诊断，便于定位降级或丢失的能力。

多重填充/描边、复杂模糊、混合模式、蒙版和混合文本样式不进入核心编辑模型。Figma 导入遇到这些能力时保留视觉结果并降级为 SVG，避免为了少量设计效果扩大运行时和属性面板复杂度。

容器节点选择以下一种布局模式：

- `free`：子节点使用世界坐标的位置和尺寸。
- `auto`：子节点遵循容器的方向、换行、间距、内边距和对齐规则；节点尺寸支持 `fixed`、`hug` 和 `fill`，并可设置最小/最大尺寸。Auto Layout 内允许显式标记绝对定位子节点。
- `grid`：子节点使用 `x`、`y`、`w`、`h` 网格坐标。

`ComponentDefinition` 保存可复用子树及其已声明的公开属性、插槽和事件。`ComponentInstance` 是轻量引用，只保存 `definitionId`、实例外部布局、公开属性覆盖、插槽内容和事件绑定，不复制完整定义树。编辑组件定义会更新它的所有实例；详细规则见：[Unity 风格组件树设计：自定义组件与实例](./2026-07-11-unity-style-component-tree-design.md#自定义组件与实例)。

`ComponentRegistry` 由宿主应用提供。每个条目具有稳定的组件 id、框架类型（`vue` 或 `react`）、实现引用、公开属性元数据、支持的事件、插槽元数据，以及可选的编辑器预览契约。

注册表还可以声明 Figma 组件映射：使用 Figma library key、组件名称和变体属性匹配业务组件，并将 Figma 变体转换为 Vue/React props。无法匹配的组件实例降级为普通节点树或 SVG，同时产生可见的导入诊断。

## 布局

Grid 容器参考 GridStack 交互：拖放插入、网格单元吸附、调整尺寸手柄、碰撞处理、嵌套 Grid，以及在移动、删除和缩小时自动紧凑排列。每个 Grid 独立设置列数；12 列只是常用预设，而非平台限制。锁定的 Grid 子项在自动重排时保持位置。

核心版本面向桌面端。一个文档只有一套桌面布局，不提供按断点覆盖的布局数据。

## 数据与动作

绑定使用结构化引用，而不是可执行的 JavaScript。它们可以引用：

- `pageState`：文档范围内可读写的变量。
- `dataSources`：由宿主应用提供的接口结果。
- `globalState`：宿主应用显式暴露的全局状态切片。

事件引用动作链。一个动作链可以更新页面状态、调用宿主动作或请求刷新宿主提供的数据。网络请求、身份认证、授权、缓存、重试和传输错误仍由宿主应用负责。

运行时遇到缺失的注册组件、无效绑定路径或不可用数据源时，必须渲染可检查、可定位的错误占位，而不是静默失败。

## 渲染架构

核心编辑器采用 DOM 内容层与 SVG 编辑覆盖层，完全由文档模型和统一的世界坐标变换控制。

- DOM 内容层负责基础节点、布局容器和真实业务组件，尽量保持编辑预览与宿主运行时一致。
- SVG 覆盖层负责选择轮廓、控制柄、多选框、吸附参考线和编辑反馈，不参与运行时页面输出。
- 两层共享同一个视口和世界坐标变换，确保缩放和平移时保持对齐。
- `RenderBackend` 定义可替换的编辑渲染后端。高级矢量扩展可以按需提供 Canvas/WebGL 后端，但核心编辑器不依赖它。

大屏运行时、页面编译器、DOM Islands、Worker 数据平面、GPU Layer、性能指标和降级策略见：[大屏与高性能架构设计](./2026-07-11-high-performance-large-screen-architecture-design.md)。

运行时严格使用 `PageDocument.runtime.adapterId` 指定的唯一 `ComponentAdapter`。Vue 和 React 是首批官方适配器；其他宿主框架可以提供兼容实现。

## 资源与字体

`AssetManifest` 管理文档引用的图片、SVG、字体和其他二进制资源。持久化文档只能保存稳定的资源 id 或宿主资源地址，不得保存临时 Blob URL、Figma 内部地址或未经处理的外部引用。

导入图片时，编辑器将内容交给宿主资源适配器存储并取得稳定资源 id。导入文本时检查字体可用性；字体缺失时保留原字体信息、使用明确的替代字体预览，并在节点和导入报告中显示警告。

SVG 导入前必须移除脚本、事件处理器和不允许的外部资源引用。无法安全解析的内容不得进入页面模型。

## Figma 导入与粘贴

第三阶段提供两条相互独立的 Figma 导入链路：

1. 标准 SVG 粘贴：用户在 Figma 中使用 Copy as SVG，然后在无限工作区中粘贴到当前画板。该链路优先保持视觉效果，结果导入为 SVG 或可解析的矢量节点，不承诺保留 Auto Layout、组件实例、变量和设计系统语义。
2. 结构化粘贴：通过配套 Figma 插件读取当前选择，将节点树、布局、样式、文本、资源和组件信息转换为版本化的 `FigmaImportPayload`，再复制到 ComposeUI 或发送给 ComposeUI 导入。该链路优先保持可编辑结构。

普通 Figma 剪贴板中的私有格式不作为稳定集成契约。结构化导入只接受 ComposeUI 声明并校验过的载荷版本；载荷必须设置节点数、树深、文本大小和资源总量限制。

### 节点转换

- Figma Page/Section 不直接成为运行时页面；用户选择的 Frame 或节点集合成为一次导入的根节点。
- Frame 转换为自由布局或 Auto Layout 容器；Group 转换为普通分组；Rectangle、Ellipse、Line、Text、Image 和 Vector 转换为对应基础节点。
- Auto Layout 转换方向、换行、间距、内边距、对齐、`fixed/hug/fill`、最小/最大尺寸和绝对定位子节点。
- Component/Instance 优先通过 `ComponentRegistry` 的 Figma 映射转换为业务组件；无法匹配时降级为普通节点树或 SVG。
- 布尔运算、蒙版、复杂矢量和当前版本不支持的效果优先降级为可显示的 SVG，并在导入报告中记录不可编辑部分。
- Figma 变量和样式先保存来源信息和解析后的静态值；除非宿主显式配置映射，否则不自动转换为页面变量或全局状态。

### 粘贴交互

- 默认粘贴到鼠标所在的世界坐标；鼠标不在画布内时粘贴到当前视口中心。
- 粘贴目标优先是当前编辑上下文中的激活容器；不兼容时退回当前页面根节点并说明原因。
- 保留选区内部的父子关系、相对坐标和绘制顺序，为所有节点生成新的本地 id，并保存外部来源 id 以供诊断。
- 整次导入是一个原子命令，可以一次撤销或重做。导入成功后选中新生成的根节点，并展示导入摘要。
- 导入报告区分“完整转换”“降级转换”和“未导入”，并能定位到对应节点。部分失败不得导致整个文档损坏。

## 矢量能力路线图

第一阶段包括基础矩形、圆形、线条、文本、图片和 SVG 展示。数据模型预留路径节点类型和布尔运算接口，供后续扩展。

高级矢量创作作为可选扩展，引入钢笔路径编辑、锚点和控制柄编辑、复合路径、布尔运算、SVG 导入/导出和复杂矢量分组。它不属于前三阶段的核心产品交付承诺，可以独立排期和安装。

## 命令、校验与测试

项目采用 Vitest 单元/属性测试、可审查黄金文件、Playwright E2E/视觉回归和多浏览器测试。完整测试分层、Golden 更新规则、Yjs 双客户端和大屏性能基准见：[测试与黄金文件策略](./2026-07-11-testing-and-golden-files-strategy.md)。

产品示例、开发调试和浏览器测试统一使用单一 Vite Playground，不引入 Storybook。场景模型、Adapter 示例、测试 API 和部署方式见：[Vite Playground 示例与演示设计](./2026-07-11-vite-playground-demo-design.md)。

所有文档变更都必须通过命令系统执行，使撤销/重做、持久化、面向协作的历史记录和确定性测试共用同一机制。

第一阶段的命令和事务必须输出稳定 Patch，并保留类型化事务来源，以支持后续 Yjs 事务桥接。协同模式下 Y.Doc 是共享权威状态，规范化 Record Store 是本地查询和渲染投影；详细设计见：[Yjs 实时协同设计](./2026-07-11-yjs-collaboration-design.md)。

在创建、移动、绑定、保存和渲染节点时执行校验。编辑器阻止不兼容的嵌套和无效的 Grid 跨度；对无法解析的注册组件元数据和无效绑定提供可操作的诊断信息。

自动化测试应覆盖文档 Schema 迁移和校验、命令撤销/重做不变量、布局计算和 Grid 碰撞场景、组件定义/实例的同步传播、绑定解析，以及 DOM 内容层与 SVG 覆盖层的坐标对齐。端到端测试应覆盖：用户将已注册组件拖入 Grid、绑定到宿主数据、保存 JSON 文档，并在宿主运行时再次渲染。

嵌入测试应覆盖 React、Vue 和原生 Web 宿主中的挂载/卸载、受控文档更新、只读模式、保存失败、编辑器 Shadow DOM 与业务组件 Light DOM 的样式边界、快捷键焦点边界、编辑器/业务组件 Portal 定位、页面级上下文继承、跨框架组件拒绝、事件监听清理，以及同一宿主页面中的多个编辑器实例。

Figma 导入测试应使用固定夹具覆盖 SVG 粘贴、结构化节点粘贴、嵌套 Auto Layout、图片和字体资源、混合文本、组件映射、未知节点降级、恶意 SVG 清理、超限载荷拒绝，以及一次撤销整次导入。转换测试必须校验节点层级、局部坐标、绘制顺序和视觉关键属性，而不能只比较生成节点数量。

## 交付顺序

1. 可嵌入内核和编辑器：包边界、宿主 API、Schema、命令历史、无限工作区、页面画板、组件树、选区、自由/Auto/Grid 布局、隔离机制和多实例支持。
2. 组件与运行时平台：框架无关的 `ComponentAdapter`、组件面板、Vue/React 官方适配器、自定义组件定义和实例、宿主数据/状态/动作绑定、诊断、文档持久化和预览模式。
3. Figma 互操作：安全 SVG 粘贴、配套 Figma 插件、结构化载荷、节点转换、组件映射、资源导入、转换诊断和导入集成测试。
4. Yjs 实时协同：Y.Doc 映射、Provider 端口、离线同步、Awareness、远程选区、本地撤销和冲突诊断。

高级矢量创作不在核心交付顺序中，作为独立可选扩展评估。
